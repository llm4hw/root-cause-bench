The problem is that out1 is being updated with a blocking assignment (using â€œ=â€ instead of â€œ<=â€) in a sequential, clocked process and is not being assigned in a fully consistent way across all branches of the reset/clocked logic. In a sequential always block with asynchronous reset, every register should be updated with nonblocking assignments so that the synthesis tool can properly infer the register with its reset. In your code, you update data with nonblocking assignment in the reset branch but then use a blocking assignment for both data (in the else branch) and out1. Because out1 is assigned outside of the reset branch (and with the wrong assignment operator), the synthesis tool has trouble extracting a proper asynchronous reset for the out1 register.

To fix the bug, update all register assignments using the nonblocking assignment operator (<=) and assign the reset value for out1 in the reset branch. For example, you can rewrite the always block as follows:

-------------------------------------------------
always @(posedge clk or posedge rst)
begin
    if (rst) begin
        data <= 1'b0;
        out1 <= 1'b0;
    end
    else begin
        data <= ~data;
        out1 <= data;
    end
end
-------------------------------------------------

This ensures that out1 is correctly updated synchronously (and reset synchronously as intended) and prevents the synthesis error regarding asynchronous set/reset extraction.
